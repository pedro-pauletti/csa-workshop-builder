Index design & ingestion

An index is a typed schema — fields, analyzers, vector configurations — that determines what you can search, filter, and rank on. Get this right and the rest of the system gets easier.

Anatomy of an index

An index is a collection of documents with a fixed schema. Each field has a type (Edm.String, Edm.Collection(Edm.Single) for vectors, …) and a set of attributes: searchable, filterable, facetable, sortable, retrievable, and key.

Common mistakes

  • Embedding the wrong field. Embed text that carries meaning (title, description). Don't embed SKU codes or opaque IDs.
  • Chunking too late. A 50-page PDF embedded as one vector is useless. Chunk into ~500-token passages before vectorising.
  • Mismatched models. Embed at ingest and at query with the same model. Mixing breaks similarity.
  • No filter fields. Without filterable attributes you cannot scope by category, price band, or tenant. Plan filters before ingest.
  • Forgetting facets. Faceted UIs (left rail, "Brand · Size · Color") need facetable from day one.
