Vector & hybrid search

Keyword search is precise but brittle. Vector search is fuzzy but recall-heavy. Hybrid runs both in parallel and fuses results — usually the right default for grounded-AI applications.

What "vector search" actually means

Each searchable text is passed through an embedding model (e.g. text-embedding-3-large) which produces a fixed-length numeric vector. Texts with similar meaning land near each other in vector space. At query time the user's question is embedded with the same model and the index returns the nearest vectors by cosine similarity, using a graph structure (HNSW) for sub-millisecond ANN lookup.

Vector vs keyword (BM25)

  • Keyword (BM25) rewards exact term matches and is unbeatable for product IDs, model numbers, and acronyms.
  • Vector rewards semantic similarity, so "warm jacket" matches "insulated parka" even if the term warm is nowhere in the document.
  • Hybrid runs both queries against the same index and fuses the result lists with Reciprocal Rank Fusion (RRF). It is the recommended default.
When to pick vector-only. Almost never for production. Almost always for a hackathon demo where the data is small and the queries are paraphrases. In real customer engagements, default to hybrid + semantic ranker.
How a hybrid query is fused

Reciprocal Rank Fusion is rank-based, not score-based — it does not require normalising disparate score scales.

What "semantic similarity" feels like

Embedding models map text to a high-dimensional space where meaning is geometry. Two sentences with similar intent end up close together, regardless of whether they share words. The demo below uses pre-computed cosine similarities so you can feel it without an API call.
